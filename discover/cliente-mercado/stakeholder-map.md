# Stakeholder Map (Mapa de Stakeholders)

## 📖 ¿Qué es?

El **Stakeholder Map** es una herramienta visual que identifica y clasifica a todas las personas, grupos o entidades que tienen interés o influencia en un proyecto, producto o iniciativa. Es fundamental para cualquier esfuerzo de diseño porque帮助你 entender quién tiene poder, quién tiene interés, y cómo se relacionan entre sí.

La técnica fue desarrollada originalmente en la gestión de proyectos y la teoría de stakeholders (Freeman, 1984), pero se ha convertido en esencial en Design Thinking porque reconoce que los productos y servicios no existen en un vacío — están inseridos en un ecosistema de personas con diferentes motivaciones, necesidades y niveles de influencia.

Un stakeholder puede ser interno (empleados, accionistas, equipo) o externo (usuarios, reguladores, competidores, comunidad). El Stakeholder Map te ayuda a ver más allá del usuario final.

## 🔧 Cómo se usa

### Paso a paso:

1. **Brainstorming inicial** — Lista todos los stakeholders potenciales (no te limites)
2. **Investiga** — ¿Quiénes son? ¿Qué les importa? ¿Qué poder tienen?
3. **Clasifica** — Por influencia y poder usando la matriz
4. **Identifica relaciones** — ¿Cómo se relacionan entre sí?
5. **Define estrategia** — Qué hacer con cada grupo
6. **Revisa periódicamente** — Los stakeholders pueden cambiar

### La Matriz de Poder/Interés:

```
                        INFLUENCIA
                    ┌───────────┬───────────┐
                    │  MEDIA    │   ALTA    │
        ┌───────────┼───────────┼───────────┤
    ALTA│  KEEP     │  MANAGE   │           │
        │  SATISFIED│  CLOSELY  │           │
        ├───────────┼───────────┼───────────┤
    P   │           │  MONITOR  │   LOW     │
    O   │   LOW     │           │  KEEP     │
    W   │  INTEREST │           │  INFORMED │
    E   ├───────────┼───────────┼───────────┤
    R   │  LOW      │   MEDIA   │           │
        │           │           │           │
        └───────────┴───────────┴───────────┘
                    INTERÉS
```

### Estrategias por cuadrante:

| Cuadrante | Stakeholders | Estrategia |
|-----------|-------------|------------|
| **Manage Closely** | Alta influencia + Alto interés | Involucrar activamente, comunicar seguido, buscar input |
| **Keep Satisfied** | Alta influencia + Bajo interés | Satisfacer necesidades, mantener satisfechos, evitar conflictos |
| **Keep Informed** | Baja influencia + Alto interés | Informar regularmente, escuchar feedback, reconocer contribuciones |
| **Monitor** | Baja influencia + Bajo interés | Observar, mínimo esfuerzo, revisar periódicamente |

### Consejos:

- **No olvides los "invisibles"** — Reguladores, medios, competidores indirectos
- **Considera relaciones** — ¿Quién está alineado con quién? ¿Quién tiene conflictos?
- **Usa dados reales** — No specules, investiga
- **Reconoce que los stakeholders cambian** — Revisa el mapa periódicamente

## 💡 Ejemplo de uso

### Contexto:
Una empresa de salud quiere lanzar una app de telemedicina. Necesitan entender el ecosistema antes de diseñar.

### Stakeholder Map creado:

```
                        INFLUENCIA
                    ┌───────────┬───────────┐
                    │  MEDIA    │   ALTA    │
        ┌───────────┼───────────┼───────────┤
    ALTA│           │  MANAGE   │           │
        │           │  CLOSELY  │           │
        ├───────────┼───────────┼───────────┤
    P   │  KEEP     │  MONITOR  │   LOW     │
    O   │  SATISFIED│           │  KEEP     │
    W   │           │           │  INFORMED │
    E   ├───────────┼───────────┼───────────┤
    R   │  LOW      │   MEDIA   │           │
        │  INTEREST │           │           │
        └───────────┴───────────┴───────────┘
                    INTERÉS

Clasificación detallada:
- MANAGE CLOSELY: Pacientes, Médicos, Equipo de producto
- KEEP SATISFIED: Inversionistas, Proveedores de tecnología
- KEEP INFORMED: Familiares de pacientes, Seguros médicos
- MONITOR: Medios, Comunidades online de salud
```

### Análisis derivado:

**Relaciones identificadas:**
- Los médicos son los "gatekeepers" — sin ellos, la app no funciona
- Los seguros pueden ser aliados o barreras
- Los reguladores (FDA/COFEPRIS) tienen poder alto aunque interés bajo

**Estrategia resultante:**
1. **Prioridad 1:** Diseñar primero para médicos (sus necesidades = adopción)
2. **Prioridad 2:** Asegurar cumplimiento regulatorio desde el inicio
3. **Prioridad 3:** Integrar con seguros después de validar el modelo

## 📅 Cuándo usarlo

### Fases recomendadas:

- **Discover** — Para mapear el ecosistema y entender relaciones
- **Define** — Para validar que el problema considera a todos los stakeholders
- **Develop** — Para diseñar soluciones que consideran todas las partes
- **Deliver** — Para planificar el launch y comunicación

### Situaciones ideales:

- ✅ Al inicio de cualquier proyecto o producto nuevo
- ✅ Para planificación de comunicación
- ✅ En proyectos con múltiples dependencias internas/externas
- ✅ Cuando hay conflictos de interés potenciales
- ✅ En productos con ecosistema complejo (saas, marketplace, plataformas)

### No usar cuando:

- ❌ El proyecto es muy simple y tiene pocos stakeholders
- ❌ No tienes información sobre los stakeholders (investiga primero)
