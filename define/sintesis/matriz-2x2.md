# Matriz 2x2 (2x2 Matrix)

## 📖 ¿Qué es?

La **Matriz 2x2** es una herramienta de síntesis visual que clasifica elementos en dos dimensiones con dos valores cada una. Es una de las herramientas más versátiles y útiles en Design Thinking porque permite priorizar, segmentar, y tomar decisiones de manera visual y clara.

La clave de la matriz 2x2 es que forza la clasificación de elementos en función de dos criterios simultáneamente, lo que revela patrones que no son obvios cuando se analizan de manera lineal. Es especialmente útil cuando hay muchos items y necesitas priorizar o tomar decisiones sobre cuáles enfocarte.

Esta herramienta tiene raíces en la gestión estratégica y la toma de decisiones, y se ha convertido en estándar en productos y diseño de servicios.

---

## 🔧 Cómo se usa

### Estructura básica:

```
                    EJE Y (Dimensión 2)
                        │
              ALTO      │       ALTO
           ┌────────────┼────────────┐
           │            │            │
           │    ★★      │    ★★★     │
           │            │            │
    BAJO ──┼────────────┼────────────┼── BAJO
           │            │            │
           │     ★      │    ★★      │
           │            │            │
           └────────────┼────────────┘
                        │
                       EJE X (Dimensión 1)
```

### Matrices comunes y sus usos:

| Matriz | Eje X | Eje Y | Uso |
|--------|-------|-------|-----|
| **Impacto-Esfuerzo** | Esfuerzo | Impacto | Priorizar features |
| **Urgencia-Importancia** | Urgencia | Importancia | Gestion de tareas |
| **Valor-Complejidad** | Complejidad | Valor | Decisiones de producto |
| **Satisfacción-Necesidad** | Necesidad | Satisfacción | Roadmap de producto |

### Paso a paso:

1. **Define las dimensiones** — Qué dos criterios vas a usar
2. **Lista los items** — Features, problemas, decisiones, etc.
3. **Evalúa cada uno** — En ambas dimensiones (escala 1-4 o alto/bajo)
4. **Posiciona cada item** — En el cuadrante correcto
5. **Toma decisiones** — Según el cuadrante donde cayeron

### Consejos:

- **Sé consistente** — Usa la misma escala para todos los items
- **Justifica las posiciones** — Por qué un item está en un cuadrante
- **Usa datos cuando sea posible** — No solo intuición
- **No te quedes con el status quo** — La matriz te da direction, no la respuesta

## 💡 Ejemplo de uso

### Contexto:
Un equipo de producto tiene 12 features en su backlog. Necesitan priorizarlas para el próximo quarter.

### Matriz Impacto-Esfuerzo creada:

```
                          IMPACTO
                      BAJO      │      ALTO
                    ┌───────────┼───────────┐
                    │           │           │
        ESFUERZO    │   LOW     │   QUICK   │
         BAJO       │   WINS    │   WINS    │
                    │           │           │
                    ├───────────┼───────────┤
                    │           │           │
        ESFUERZO    │   LOW     │   STRATE- │
         ALTO       │  PRIORITY │   GIC     │
                    │           │   BET    │
                    └───────────┴───────────┘
                          IMPACTO
                         (ALTO)
```

### Items posicionados:

| Feature | Esfuerzo | Impacto | Cuadrante | Decisión |
|---------|-----------|---------|-----------|----------|
| **A: Notificaciones push** | Bajo | Alto | Quick Win | ✅ Prioridad #1 |
| **B: Login social** | Bajo | Medio | Quick Win | ✅ Hacer pronto |
| **C: Dark mode** | Bajo | Bajo | Low Priority | ⏸️ Postergar |
| **D: Checkout redesign** | Alto | Alto | Strategic Bet | ⏭️ Para después |
| **E: Analytics dashboard** | Alto | Medio | Low Priority | ❌ No hacer |
| **F: Onboarding flow** | Medio | Alto | Quick Win | ✅ Hacer pronto |

### Decisión tomada:
> "Priorizamos las Quick Wins (notificaciones, login social, onboarding) para el próximo sprint. El checkout redesign es importante pero requiere más recursos, lo movemos a Q3."

## 📅 Cuándo usarlo

### Fases recomendadas:

- **Define/Síntesis** — Para priorización de problemas
- **Develop** — Para priorización de features y soluciones
- **Deliver** — Para decisiones de roadmap

### Situaciones ideales:

- ✅ Para validar scope de proyecto
- ✅ En roadmap planning
- ✅ Para tomar decisiones de diseño
- ✅ Cuando hay muchos items y necesitas filtrar
- ✅ En workshops con stakeholders

### No usar cuando:

- ❌ Tienes muy pocos items (overkill)
- ❌ Las dimensiones no están claras (define primero)
- ❌ Es solo para show (debe conducir a decisiones reales)
