# Roadmap de Prototipado (Prototyping Roadmap)

## 📖 ¿Qué es?

El **Roadmap de Prototipado** es un plan visual que muestra la secuencia y timeline de los prototipos a crear, desde la idea hasta el producto final. Define qué se prototipa, cuándo, y con qué nivel de fidelidad.

El roadmap te ayuda a entender el viaje completo: no todas las prototipos son iguales, y cada fase tiene un propósito diferente. Algunos prototipos son para aprender, otros para validar, y otros para comunicar.

Este roadmap es especialmente útil cuando trabajas en proyectos complejos con múltiples features y un timeline ajustado. Te ayuda a priorizar qué prototipar primero y qué puede esperar.

---

## 🔧 Cómo se usa

### Estructura típica:

```
ROADMAP DE PROTOTIPADO
═══════════════════════════════════════════════════════════

Fase 1: Exploración (Semana 1-2)
├── Prototipo A: Sketch en papel
└── Prototipo B: Wireframe básico

Fase 2: Validación (Semana 3-4)
├── Prototipo C: Clickable mid-fi
└── Prototipo D: User testing

Fase 3: Refinamiento (Semana 5-6)
├── Prototipo E: High-fi mockup
└── Prototipo F: Prototipo funcional

Fase 4: Lanzamiento (Semana 7-8)
└── MVP: Producto mínimo viable
```

### Criterios para definir el roadmap:

| Criterio | Pregunta | Por qué importa |
|----------|----------|------------------|
| **Prioridad de features** | ¿Cuál feature es más crítica? | Primero lo más importante |
| **Riesgo** | ¿Qué tiene mayor incertidumbre? | Lo que más necesita testing |
| **Recursos** | ¿Cuánto tiempo y personas tenemos? | Define scope realista |
| **Dependencias** | ¿Qué debe terminarse antes? | Para evitar bloqueos |

### Cómo crear el roadmap:

1. **Lista todos los features** — Del backlog o specs
2. **Define la fidelidad necesaria** — Baja/media/alta por feature
3. **Identifica dependencias** — Qué necesita qué
4. **Asigna timeline** — Semanas disponibles
5. **Prioriza** — Qué va primero
6. **Comunica** — Comparte con el equipo

### Consejos:

- **Comienza con low-fi** — No necesitas high-fidelity desde el inicio
- **Itera** — El roadmap evoluciona con learnings
- **Allocate buffer** — Para iterar y pivotar
- **Comunica cambios** — Si algo se desplaza, avisa

## 💡 Ejemplo de uso

### Contexto:
Un equipo de 2 diseñadores tiene 6 semanas para prototipar una nueva feature de checkout para una app.

### Roadmap creado:

```
ROADMAP DE PROTOTIPADO - CHECKOUT FEATURE
═══════════════════════════════════════════════════════════

Semana 1-2: EXPLORACIÓN ⚡
├── Prototipo 1: Paper prototype del flujo completo
│   ├── Meta: Validar estructura del flujo
│   └── Fidelidad: Low (sketches)
├── Prototipo 2: Wireframes de cada pantalla
│   ├── Meta: Definir layout y contenido
│   └── Fidelidad: Low-mid
└── Entregable: Flujo aprobado por product

Semana 3: VALIDACIÓN 🧪
├── Prototipo 3: Clickable prototype (Figma)
│   ├── Meta: Testear con usuarios
│   └── Fidelidad: Mid-fi
└── Testing: 5 usuarios, tareas de checkout

Semana 4: REFINAMIENTO 🔧
├── Prototipo 4: High-fidelity mockup
│   ├── Meta: Detalle visual completo
│   └── Fidelidad: Alta
├── Prototipo 5: Prototype funcional (Principle/ProtoPie)
│   ├── Meta: Probar interacciones
│   └── Fidelidad: Alta
└── Entregable: Design specs para dev

Semana 5-6: ITERACIÓN 🔁
├── Revisión de stakeholders
├── Ajustes basada en feedback
├── Documentación final
└── Handoff a desarrollo
```

### Priorización aplicada:

| Feature | Prioridad | Fase | Razón |
|---------|-----------|------|-------|
| Flujo de compra | 🔴 Alta | Sem 1-2 | Core del producto |
| Métodos de pago | 🔴 Alta | Sem 3-4 | Depende del flujo |
| Guest checkout | 🟡 Media | Sem 4 | Enhancement |
|Guardar métodos | 🟢 Baja | Sem 5-6 | Nice to have |

## 📅 Cuándo usarlo

### Fases recomendadas:

- **Develop** — Para planificación de prototipado
- **Deliver** — Para handoff a desarrollo

### Situaciones ideales:

- ✅ Para planificar el trabajo de diseño
- ✅ Para comunicar timeline al equipo
- ✅ Para identificar bottlenecks
- ✅ Para alinear expectativas con stakeholders

### No usar cuando:

- ❌ El proyecto es muy simple (overkill)
- ❌ Tienes una sola iteración (prototipo único)
